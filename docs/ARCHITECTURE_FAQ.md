# Architecture FAQ

本文聚焦 EIT-DB 的核心架构取舍，回答常见设计问题。

## 1. 为什么采用 Adapter 模式，而不是 Active Record 风格？

1. EIT-DB 的目标是多数据库能力治理，不是单后端 ORM 简化层。
2. Adapter 模式可将能力声明、降级策略、运行时体检放在统一边界。
3. Query IR + Compiler 需要与模型层解耦，避免业务模型承载方言复杂度。

## 2. 为什么强调“显式差异”，而不是“数据库无差别抽象”？

1. 不同数据库能力客观不等价，强行抹平会导致业务层出现隐性补丁。
2. 显式差异 + 可控降级更利于行为可解释、可审计、可观测。
3. 这对生产故障定位和长期维护成本更友好。

## 3. 为什么图数据库与图能力放在第一梯队支持？

1. 社交、推荐、风控、关系分析本质上是图问题。
2. 如果后置图能力，应用层会过早固化关系补丁并增加迁移成本。
3. 第一梯队支持可尽早建立关系语义边界与统一查询路径。

## 4. 为什么 Adapter 放在主仓，而不是立即拆成子项目？

1. 早期阶段需要保证能力矩阵、回退策略、编译边界一致演进。
2. 主仓协作可减少版本漂移和跨仓兼容回归成本。
3. 在边界稳定后再按成熟度拆分 Provider 更稳妥。

## 5. 为什么选择 Go，而不是 Java/C#/Rust？

1. Go 在基础设施组件场景具备较低部署和运维成本。
2. 对协作层这类并发 IO 场景，Go 的并发模型足够高效。
3. 项目优先目标是工程可扩展性与交付效率平衡，而非单点极限性能。

## 6. 协作层是不是强制开启？

不是。

1. 单适配器业务可只使用一个 `Repository`。
2. 只有跨适配器调度、链路追踪、监控树需求才建议启用协作层。
3. 协作层按能力视图组合接入，不要求全局强绑定。

## 7. 为什么保留 fallback，不强制原生能力？

1. 生产环境能力差异（版本、插件、权限）很常见。
2. fallback 保障可用性，避免能力缺失直接导致核心流程中断。
3. 同时通过 reason 和体检报告维持可观测，避免静默退化。

## 8. Redis + Arango 协作层在架构中的定位是什么？

1. Redis：实时消息与管理控制面（节点/分组/心跳）。
2. Arango：持久账本与在线状态投影（审计与追踪）。
3. 业务数据库：实际查询与写入执行面。

## 9. 为什么在调度设计中引入 blueprint_tick / route_tick？

1. 仅依赖 adapter_id 会在扩缩容/重建时导致不可控漂移。
2. `blueprint_tick` 表示规则版本，`route_tick` 表示候选快照版本。
3. 分层 tick 提升路由稳定性与追溯能力。

## 10. 如何判断项目是否适合 EIT-DB？

适合：

1. 多数据库并存且能力差异明显。
2. 需要能力治理、体检门禁、可控降级。
3. 需要长期演进的查询编译与调度框架。

不太适合：

1. 单库简单 CRUD、短周期 MVP。
2. 团队更偏好极简 ORM 心智与最低配置成本。

## 11. 未来会拆分独立 Provider 吗？

会在边界稳定后逐步推进，原则：

1. 保持主仓能力矩阵与协议一致性。
2. 以成熟度和回归成本为门槛，而不是先拆再治理。

## 12. ArangoDB 的 JOIN 能力应当与 Neo4j 视为一致吗？

可以一致，但要区分统计口径。

1. 在图关系语义层（关系遍历、图关联）上，两者都属于 first-class 支持。
2. 在 SQL JOIN 统计口径上，两者都不是“传统 SQL JOIN 全支持”模型。
3. 文档中应同时标注这两个层次，避免把“SQL JOIN 缺失”误读为“图关系能力弱”。

## 13. 为什么文档口径要从 JOIN 覆盖率转为关系语义支持等级？

1. 项目将进入后端无关化 API 改造阶段，用户不应被迫理解每个后端的语法入口差异。
2. 相比 JOIN 语法覆盖率，关系语义支持方式（图原生 / 外键连接 / 聚合模拟）更能反映真实建模能力。
3. 统一分级后，Blueprint 路由与 Schema 关系编译可以围绕语义做决策，而不是围绕单一方言特性做决策。
