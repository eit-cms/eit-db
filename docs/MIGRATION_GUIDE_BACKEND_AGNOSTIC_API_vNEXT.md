# 后端无关 API 迁移指南（vNext 草案）

本文给出从“后端专属执行入口”迁移到“统一 Query/Exec + 泛事务语义 + 关系语义优先”的建议路径。

适用对象：

1. 已在业务代码中直接使用后端专属入口（如 Cypher/AQL 显式入口）。
2. 准备进入 Blueprint + 显式 Schema 关系编译阶段。
3. 希望减少后端分支判断和 API 心智负担。

## 1. 迁移目标

1. 统一执行入口：业务侧优先使用统一 Query/Exec 语义入口，由适配器映射到 SQL/Cypher/AQL/聚合计划。
2. 统一事务语义：凡是支持多语句原子提交的后端，统一视为支持事务语义。
3. 统一关系语义：以关系语义支持方式（图原生 / 外键连接 / 聚合模拟）为主口径。
4. 统一路由基线：基于 Blueprint + 显式 Schema 关系注册驱动编译与路由。

## 2. 迁移分阶段

### 阶段 A：盘点与冻结

1. 盘点代码中后端专属 API 调用点。
2. 给这些调用点打标签：
   - 读路径
   - 写路径
   - 事务边界
   - 关系语义查询
3. 冻结新增后端分支：迁移期不再新增新的后端专属业务分支。

建议检索关键字：

- QueryCypher
- ExecCypher
- ExecuteAQL
- document_join
- StartSession

### 阶段 B：统一执行入口

1. 将业务层读写入口收敛到统一 Query/Exec 语义。
2. 后端专属执行函数保留在适配器能力层，仅作为兼容路径。
3. 所有新代码默认通过统一入口接入，减少 adapter type switch。

验收标准：

1. 业务服务层不再出现按后端分叉的大段执行逻辑。
2. 新增功能不依赖后端专属执行函数作为第一入口。

### 阶段 C：泛事务语义

1. 将“是否支持事务”从 SQL 专属概念升级为“是否支持多语句原子提交语义”。
2. 事务能力由适配器能力声明给出，不在业务层硬编码判断。
3. 业务层事务边界统一收敛到同一事务抽象入口。

验收标准：

1. 事务开关判断来自能力声明而不是后端名称。
2. 多语句写流程在支持事务语义的后端可按统一方式执行。

### 阶段 D：关系语义重构

1. 将 JOIN 覆盖率比较替换为关系语义分级：
   - 强支持：图原生关系语义（Neo4j / Arango）
   - 中支持：外键 + JOIN（SQL 后端）
   - 弱支持：聚合模拟（MongoDB）
   - 不适用：键值/流后端（Redis）
2. 查询编译围绕显式 Schema 关系而非字符串 JOIN 片段。
3. Blueprint 路由优先考虑语义能力与约束，而不是方言语法可用性。

验收标准：

1. 设计评审不再使用“某后端 JOIN 支持百分比”作为主要判断。
2. 复杂关系查询的后端选择依据变为语义能力匹配。

## 3. 风险与缓解

1. 风险：兼容期内同时存在统一入口与后端专属入口，可能导致认知混乱。
   - 缓解：新增代码默认只允许统一入口，专属入口仅用于适配器层或过渡封装。
2. 风险：事务语义在不同后端边界不一致。
   - 缓解：通过能力声明暴露限制，并在 Blueprint 约束中显式表达。
3. 风险：关系语义迁移导致历史 SQL 心智失配。
   - 缓解：提供关系语义映射示例与分级决策表。

## 4. 推荐落地顺序

1. 先完成调用点盘点与新增冻结。
2. 先迁移统一执行入口，再迁移事务边界。
3. 最后推进关系语义重构与 Blueprint 路由收敛。
